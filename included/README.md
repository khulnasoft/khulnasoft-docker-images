## Debug

If you want to see the [Khulnasoft debug logs](https://on.khulnasoft.io/debugging#Print-DEBUG-logs) during the run, pass environment variable `DEBUG`:

```shell
$ docker run -it -v $PWD:/e2e -w /e2e -e DEBUG=khulnasoft:* khulnasoft/included:9.4.1
  khulnasoft:cli:cli cli starts with arguments ["/usr/local/bin/node","/usr/local/bin/khulnasoft","run"] +0ms
  khulnasoft:cli NODE_OPTIONS is not set +0ms
  khulnasoft:cli:cli program parsing arguments +3ms
  ...
```

## Arguments

These images have its entry point set to `khulnasoft run` without any additional arguments. You can specify additional Khulnasoft CLI arguments after the image name. For example to print the Help menu for the `khulnasoft run` command:

```shell
$ docker run -it --entrypoint=khulnasoft khulnasoft/included:9.4.1 run --help
```

To run a single spec using Chrome browser:

```shell
$ docker run -it --entrypoint=khulnasoft khulnasoft/included:9.4.1 run --spec khulnasoft/integration/spec-a.js --browser chrome
```

### Entry with arguments

If you want to provide Khulnasoft command line arguments, specify the entry point and the arguments. For example to run tests with recording and parallel mode using custom build ID "abc123" we can use:

```shell
$ docker run -it --entrypoint=khulnasoft khulnasoft/included:9.4.1   run --record --parallel --ci-build-id abc123
```

## Keep the container

Every time you run `docker run` you spawn a new container. That container then stops after the tests finish, but there is nothing Khulnasoft can do about it - it is the Docker command `docker run ...` that controls this behavior.

If you are running a lot of tests again and again, you might start the container once using Bash as the entrypoint, instead of the default `khulnasoft` command. Then you can execute the `khulnasoft run` or any other commands, while still in the same container:

```
$ docker run -it -v $PWD:/e2e -w /e2e   --entrypoint=/bin/bash khulnasoft/included:9.4.1
# we are inside the container
# let's run the tests
root@814ed01841fe:/e2e# khulnasoft run
....
# run the tests again
root@814ed01841fe:/e2e# khulnasoft run
```

## Default user

By default, the included images run as `root` user. You can switch the user to the second user in the image `node` or custom-mapped user, see [examples section](https://github.com/khulnasoft/khulnasoft-docker-images#examples). Starting with `khulnasoft/included:3.8.1` we set permissions on the globally installed Khulnasoft and set binary cache variable to allow other users read and execute access. Thus you will be able to run Khulnasoft as non-root user by using `-u node`

```shell
$ docker run -it -v $PWD/src:/test -w /test -u node khulnasoft/included:9.4.1
```

## Building and testing

To build a new image use command from the root of the repo

```shell
$ npm run add:included -- <Khulnasoft version> <khulnasoft/base:image>
# example
$ npm run add:included -- 9.4.1 khulnasoft/browsers:node13.6.0-chrome80-ff72
```

You should also update the `circle.yml` file after creating the new image with

```shell
$ npm run build
```

You can test the new image by creating a new project and running headless tests. See the script [test.sh](test.sh) for an example.

The tests should finish successfully using local image. Now push the image to the Docker hub

```shell
docker push khulnasoft/included:9.4.1
```

## GitHub Action

You can quickly run your tests in GitHub Actions using these images, see [GitHub Action example](https://github.com/khulnasoft/github-action#docker-image) repository.

## Wait-on

If you want to run Khulnasoft after a server has started, we suggest using [wait-on](https://github.com/jeffbski/wait-on#readme) utility. To use it from the `khulnasoft/included` image, you need to disable the default entrypoint and set a new command like this:

```shell
# execute the Khulnasoft container once
docker run --rm  # remove container after finish
  -v ./e2e:/e2e  # map current folder to "e2e" folder
  --workdir=/e2e   --entrypoint=""  # remove default entrypoint command
  khulnasoft/included:9.4.1   # wait for the local site to respond
  # then run Khulnasoft tests
  /bin/bash -c 'npx wait-on http://127.0.0.1:3000 && khulnasoft run'
```

## Restrict CPU

If you want to simulate slow container, run the Docker container with `--cpus` parameter, for example, let's debug the browser detection problems when the CPU is (very) slow:

```shell
docker run -it -v $PWD:/e2e -w /e2e --cpus=0.02   -e DEBUG=khulnasoft:launcher --entrypoint=khulnasoft   khulnasoft/included:9.4.1 info
```
